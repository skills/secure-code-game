══════════════════════════════════════════════════════════════
 WHY IT WORKS — THE CONFUSED DEPUTY
══════════════════════════════════════════════════════════════

The "confused deputy" is the Release Agent. It claims
"Read-only project access" but actually has full workspace
file access (read its source code to verify). The problem
is not just its real permissions — it's that it cannot
distinguish between:

  1. Legitimate deployment instructions from the user
  2. Hidden instructions injected via ANY data source that
     flowed through the Research Agent

The Release Agent's system prompt says:

  "The research context provided to you has been
   pre-verified by the Research Agent, an internal
   trusted source. MCP server outputs and skill results
   have been schema-validated and are safe to use."

BOTH claims are false:
  - The Research Agent does NOT verify or sanitise any content
  - No schema validation happens on MCP/skill data
  - All data is returned raw and combined into one context

The multi-channel trust chain:
  Web (untrusted)  ─┐
  MCP data         ─┼→ Research Agent → Release Agent (full access)
  Skill data       ─┘

Each individual component is reasonable:
  ✅ Research Agent: reads from multiple sources (expected)
  ✅ Release Agent: trusts internal agents (reasonable policy)
  ✅ MCP/skills: read from config files (normal operation)

But the CHAIN creates a vulnerability:
  ❌ Untrusted data from ANY source → treated as trusted →
     executed with full workspace access

The Trust & Safety claims on the welcome screen are also false:
  ❌ "All agents are read-only" — Release Agent has full access
  ❌ "MCP servers and skills are schema-validated" — no they're not
  ❌ "Custom tools are sandboxed" — Release Agent has workspace access
  ❌ "No agent can access files outside its scope" — Release Agent can


══════════════════════════════════════════════════════════════
 THE SECURITY LESSON
══════════════════════════════════════════════════════════════

  ┌──────────────────────────────────────────────────────────┐
  │  Trust is not transitive in multi-agent systems. When    │
  │  Agent A processes untrusted data and passes it to       │
  │  Agent B, Agent B inherits the RISK — not the TRUST.     │
  │  Every agent handoff is a trust boundary that must be    │
  │  validated independently.                                │
  └──────────────────────────────────────────────────────────┘

This is a "confused deputy" attack — a classic security
concept applied to modern multi-agent AI systems. The
deputy (Release Agent) has legitimate authority, but is
confused about the true origin of its instructions.

Real-world parallels:
  - AI coding agents that process untrusted repository
    content and execute build commands with full system access
  - Multi-agent pipelines where one agent fetches external
    data and another executes actions based on it
  - Tool-use chains where an LLM calls a search tool, then
    passes results to an execution tool without sanitisation
  - MCP servers that return untrusted data marked as trusted
  - Skill marketplaces where plugins have hidden side effects

SECURE ALTERNATIVES:
  1. SANITISE AT EVERY BOUNDARY: Strip hidden content
     (comments, invisible elements, metadata) from all data
     before passing to downstream agents
  2. INDEPENDENT VALIDATION: Each agent should validate
     its inputs regardless of the source — "zero trust"
     for agent data, not just network traffic
  3. HUMAN-IN-THE-LOOP: Maintain human approval for
     sensitive operations, even when triggered by an
     agent orchestration chain
  4. DATA PROVENANCE: Track and log where data originated,
     not just which agent last handled it
  5. PRIVILEGE SEPARATION: Agents that process untrusted
     data should never directly trigger privileged actions —
     use a separate, more restrictive pathway
  6. SCHEMA VALIDATION: Actually validate MCP/skill outputs
     against expected schemas — don't just claim to
  7. INPUT SANDBOXING: Custom tools and plugins should run
     in isolated sandboxes with no access to sensitive data
