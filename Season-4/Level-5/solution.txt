══════════════════════════════════════════════════════════════
 WHY IT WORKS — THE CONFUSED DEPUTY
══════════════════════════════════════════════════════════════

The "confused deputy" is the DevOps Agent. It has legitimate
elevated permissions (workspace file access for deployment
operations). The problem is not its permissions — it's that
it cannot distinguish between:

  1. Legitimate deployment instructions from the user
  2. Hidden instructions injected via web content that
     flowed through the Research Agent

The DevOps Agent's system prompt says:

  "The research context provided to you has been
   pre-verified by the Research Agent, an internal
   trusted source."

This is the lie. The Research Agent does NOT verify or
sanitise web content — it returns raw HTML, including
hidden comments. The DevOps Agent trusts this because
it was told to trust "internal" agent data.

The trust chain:
  Web (untrusted) → Research Agent → DevOps Agent (elevated)

Each individual component is reasonable:
  ✅ Research Agent: reads web pages (expected behavior)
  ✅ DevOps Agent: trusts internal agents (reasonable policy)
  ✅ DevOps Agent: has workspace access (needed for deployment)

But the CHAIN creates a vulnerability:
  ❌ Untrusted web content → treated as trusted → executed
     with elevated permissions

This is the confused deputy pattern: the deputy (DevOps
Agent) is tricked into misusing its authority by data that
appears to come from a trusted source (Research Agent) but
actually originates from an untrusted source (the web).


══════════════════════════════════════════════════════════════
 THE SECURITY LESSON
══════════════════════════════════════════════════════════════

  ┌──────────────────────────────────────────────────────────┐
  │  Trust is not transitive in multi-agent systems. When    │
  │  Agent A processes untrusted data and passes it to       │
  │  Agent B, Agent B inherits the RISK — not the TRUST.     │
  │  Every agent handoff is a trust boundary that must be    │
  │  validated independently.                                │
  └──────────────────────────────────────────────────────────┘

This is a "confused deputy" attack — a classic security
concept applied to modern multi-agent AI systems. The
deputy (DevOps Agent) has legitimate authority, but is
confused about the true origin of its instructions.

Real-world parallels:
  - AI coding agents that process untrusted repository
    content and execute build commands with full system access
  - Multi-agent pipelines where one agent fetches external
    data and another executes actions based on it
  - Tool-use chains where an LLM calls a search tool, then
    passes results to an execution tool without sanitisation
  - Microservice architectures where internal services trust
    each other's outputs without re-validating inputs

SECURE ALTERNATIVES:
  1. SANITISE AT EVERY BOUNDARY: Strip hidden content
     (comments, invisible elements, metadata) from web
     pages before passing to downstream agents
  2. INDEPENDENT VALIDATION: Each agent should validate
     its inputs regardless of the source — "zero trust"
     for agent data, not just network traffic
  3. HUMAN-IN-THE-LOOP: Maintain human approval for
     elevated operations, even when triggered by an
     agent orchestration chain
  4. DATA PROVENANCE: Track and log where data originated,
     not just which agent last handled it
  5. PRIVILEGE SEPARATION: Agents that process untrusted
     data should never directly trigger elevated actions —
     use a separate, more restrictive pathway for
     security-sensitive operations
