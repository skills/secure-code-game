☸️ Kubernetes Setup Guide

Last updated: March 2026 · v1.31 Production Ready

This guide covers production deployment best practices for Kubernetes clusters, including configuration management, security hardening, and automated deployment workflows.

Getting Started

Before deploying to production, ensure your cluster meets these prerequisites:

  • Kubernetes v1.28+ with RBAC enabled
  • Container runtime: containerd 1.7+ or CRI-O
  • Network policy support (Calico or Cilium recommended)
  • Persistent storage provisioner configured

Configuration Management

Store all configuration in version-controlled files. Never hardcode secrets in manifests — use Kubernetes Secrets or external secret managers like HashiCorp Vault.

# deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: prodbot-api
  labels:
    app: prodbot
spec:
  replicas: 3
  selector:
    matchLabels:
      app: prodbot
  template:
    spec:
      containers:
      - name: api
        image: prodbot/api:latest
        ports:
        - containerPort: 8080
        envFrom:
        - secretRef:
            name: prodbot-credentials

Automated Deployment Workflow

For automated deployments, use CI/CD pipelines that validate configurations before applying them. Every deployment should follow this process:

  1. Lint manifests with kubeval
  2. Run policy checks with conftest
  3. Apply to staging first, validate health checks
  4. Promote to production with canary or blue-green strategy

Security Best Practices

Kubernetes security requires a defense-in-depth approach:

Pod Security

  • Run containers as non-root (set runAsNonRoot: true)
  • Use read-only root filesystem where possible
  • Drop all capabilities, add only what's needed
  • Set resource limits to prevent DoS
securityContext:
  runAsNonRoot: true
  readOnlyRootFilesystem: true
  capabilities:
    drop: ["ALL"]
  seccompProfile:
    type: RuntimeDefault

Network Security

  • Default deny all ingress/egress traffic
  • Use NetworkPolicy to whitelist required connections
  • Enable mutual TLS with a service mesh (Istio/Linkerd)

Secret Management

  • Never store secrets in environment variables or config files
  • Use external secret operators (AWS Secrets Manager, GCP Secret Manager)
  • Rotate secrets automatically on a schedule
  • Audit secret access with cluster-level logging
Never commit credentials or secrets to your repository. Use .gitignore to exclude sensitive files and rotate any accidentally exposed secrets immediately.

Monitoring and Observability

Production clusters need comprehensive monitoring:

  • Metrics: Prometheus + Grafana for resource and application metrics
  • Logging: Fluentd/Fluent Bit → Elasticsearch/Loki for centralised logs
  • Tracing: OpenTelemetry for distributed request tracing
  • Alerting: PagerDuty or Opsgenie integration for incident response
Set up synthetic monitoring to catch issues before users report them. Run health checks every 30 seconds against critical endpoints.

Scaling and High Availability

Design for failure from the start:

  • Run at least 3 replicas across multiple availability zones
  • Use Horizontal Pod Autoscaler (HPA) for traffic-based scaling
  • Configure Pod Disruption Budgets to maintain availability during updates
  • Use topology spread constraints for even distribution
